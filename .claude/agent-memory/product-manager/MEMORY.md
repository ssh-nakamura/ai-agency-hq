# Product Manager（事業開発部長）ナレッジ

## プロジェクト理解

### ShieldMe概要
- 誹謗中傷検知SaaSだが、UXの核心は「ポジティブ・ファースト」設計
- 競合は月5万円〜の企業向け。個人向け月980円〜は空白地帯
- ターゲット: マイクロインフルエンサー、VTuber、配信者（ITリテラシー高、誹謗中傷深刻）

### MVP設計の判断軸
- **MVPスコープの基準**: ユーザーが「これだけで価値を感じるか」
  - 作る: X検索、AI分類、ポジティブ表示、ネガティブ要約、AIカウンセリング
  - 作らない: 認証、決済、定期監視、他SNS対応
- **コスト制約**: Phase 1は月3万円以内。X API（$100）は固定費として必須

### 技術選定の記録
- **Streamlit採用理由**: Pythonで完結、プロトタイプに最適、リッチUIが簡単
- **Claude API**: HaikuとSonnetを使い分け。分類=Haiku（安い）、カウンセリング=Sonnet（共感力）
- **X API v2 Basic**: 月$100で10,000リクエスト。1日1回検索なら約330人まで対応可能

### 差別化ポイント（競合調査から）
1. 価格破壊: 既存の1/50
2. ポジティブ・ファースト: 既存はネガティブ検知のみ
3. AIカウンセリング: メンタルケア機能は他にない
4. セルフサービス: 営業不要、即日利用可能

## 仕様策定のパターン

### ペルソナ作成のポイント
- 名前・年齢・職業だけでなく、**課題・ITリテラシー・予算感**まで明記
- 複数ペルソナで幅を持たせる（個人/副業/事務所）

### RICE優先度の使い方
- Reach: 全ユーザーが使うか、一部か（%）
- Impact: 3=大（コア価値）、2=中、1=小
- Confidence: 100%=確実、80%=高い、50%=不確実
- Effort: 人日で見積もる
- **100点満点のスコアはコア機能の証**（AI分類、ネガティブ要約）

### MVP vs Phase 2の切り分け
- Must Have: なければプロダクトが成立しない
- Should Have: あった方がいいが、なくてもローンチ可能
- 判断に迷ったら「これがなくてもユーザーの核心的課題は解決できるか？」→ Yes = Phase 2

## 未決定事項の管理
- 選択肢、判断に必要な情報、期限を明記
- 開発チームに丸投げしない（「要検証」は具体的に何を検証するか書く）
