# 仕様書作成

## 引数
- $ARGUMENTS: 対象（例: "ShieldMe通知機能", "ユーザー管理画面"）
  - 引数なし → CEOが対象を決めてから開始

## ワークフロー

### Step 1: スコープ定義（CEO）
- 何の仕様か（新機能 / 改修 / MVP）
- 対象プロダクト
- 優先度（Must / Should / Could）
- 既存仕様書があれば参照パス

### Step 2: 仕様書作成（product-manager）
product-managerをTask起動:

```
対象: {対象}
プロダクト: {プロダクト名}
既存仕様: {あればパス}

以下の構成で仕様書を書け:
1. 概要（1段落で何を作るか）
2. ユーザーストーリー（「〜として、〜したい。なぜなら〜」形式、3つ以上）
3. 機能要件（Must / Should / Could で分類）
4. 非機能要件（パフォーマンス、セキュリティ、アクセシビリティ）
5. データモデル（必要なら）
6. APIエンドポイント（必要なら）
7. UI/UXの方針（ワイヤーフレーム的な説明）
8. テスト方針（何をどうテストするか）
9. 制約・前提条件
10. 未決事項（決まっていないこと）

成果物: docs/specs/{slug}-spec.md
```

### Step 3: レビュー（CEO + 関係者）
- ユーザーストーリーは実際のユースケースに基づいているか
- 技術的に実現可能か
- 未決事項が放置されていないか

## 出力
成果物: `docs/specs/{slug}-spec.md`
